# Solid Islands with Jinja2 and JX

Use this reference when a server-rendered Jinja2 or JX page needs isolated
client-side interactivity without turning the whole page into a SPA.

This pattern is an integration guideline for this repo style. The Solid docs
confirm the relevant building blocks and boundaries:

- SolidStart exposes `experimental.islands` in `defineConfig`
  (<https://docs.solidjs.com/solid-start/reference/config/define-config>).
- Solid Router documents that islands or partial hydration solutions have
  context boundaries, so shared router context should not be assumed across
  non-hydrating regions
  (<https://docs.solidjs.com/solid-router/guides/migration>).
- Solid's `<NoHydration>` renders on the server but skips client hydration for
  its children
  (<https://docs.solidjs.com/reference/components/no-hydration>).

## When to Use

- A Jinja2 or JX page is primarily server-rendered.
- Only small parts of the page need live state, events, tabs, filters, or
  dialogs.
- You want to preserve server-owned HTML, SEO, and template composition.

## Recommended Model

Use Jinja2 or JX to own the page shell, data rendering, and static layout.
Mount Solid only into explicit island roots for interactive widgets.

Do this:

```text
Jinja/JX owns page -> emits island root -> Solid mounts widget into root
```

instead of this:

```text
Jinja/JX renders placeholder -> client re-renders entire page as app
```

## Mount Point Contract

The server template should emit a stable element for each island plus any data
the island needs.

Jinja or JX example:

```jinja
<section class="account-panel">
  <h2>{{ title }}</h2>
  <div data-island="account-menu">
    <script type="application/json" data-island-props>
      {{ island_props | tojson }}
    </script>
  </div>
</section>
```

Solid bootstrap example:

```tsx
import { render } from 'solid-js/web'

import { AccountMenu } from './islands/AccountMenu'

for (const element of document.querySelectorAll('[data-island="account-menu"]')) {
  const scriptTag = element.querySelector('script[data-island-props]')
  const props = scriptTag ? JSON.parse(scriptTag.textContent ?? '{}') : {}

  render(() => <AccountMenu {...props} />, element)
}
```

*Note: For simple scalar values, `data-props` attributes are fine, but for complex or large JSON payloads, embedding a `<script type="application/json">` directly inside the island root avoids HTML quote escaping issues and improves parsing performance.*

### Injecting the Script in Jinja

For the islands to hydrate, the server-rendered template must explicitly include the compiled JavaScript entry point. Because you are using modern tooling (like Vite), include the script as a `module`.

```jinja
{# In your base layout or specific page template #}
<section class="account-panel">
  ...
</section>

{% block scripts %}
  <!-- Example static path if serving the compiled dist/ directory -->
  <script type="module" src="/static/dist/entry-dashboard.js"></script>

  <!-- Or, if using a Vite-Jinja integration/manifest parser: -->
  <!-- {{ vite_asset('src/entry-dashboard.tsx') }} -->
{% endblock %}
```

### The Entry Point File (`entry-dashboard.tsx`)

Your Vite entry point file is responsible for finding all the island placeholders generated by Jinja and rendering the Solid components into them.

```tsx
// src/entry-dashboard.tsx
import { render } from 'solid-js/web'

// Import all the islands needed for this specific page/bundle
import { AccountMenu } from './islands/AccountMenu'
import { SetupWizard } from './islands/SetupWizard'

// Helper function to mount islands dynamically
function mountIsland(name: string, Component: any) {
  const elements = document.querySelectorAll(`[data-island="${name}"]`)

  for (const element of elements) {
    const scriptTag = element.querySelector('script[data-island-props]')
    const props = scriptTag ? JSON.parse(scriptTag.textContent ?? '{}') : {}

    render(() => <Component {...props} />, element)
  }
}

// Map the HTML data-island names to their Solid components
mountIsland('account-menu', AccountMenu)
mountIsland('setup-wizard', SetupWizard)
```

## With JX Components

In JX, prefer a component that owns the server markup and reserves a slot for
the island root.

```jinja
{# def title, island_props #}
<section class="card">
  <h2>{{ title }}</h2>
  <div
    data-island="filters"
    data-props='{{ island_props | tojson }}'
  ></div>
</section>
```

Use JX for:

- page composition;
- named slots;
- HTML structure and semantics;
- server-rendered defaults and fallbacks.

Use Solid for:

- local interactive state;
- event handling;
- optimistic UI;
- client-only widgets that should not leak into template logic.

## SolidStart Islands

If the frontend is already using SolidStart, you can also enable islands mode in
`app.config.ts`:

```ts
import { defineConfig } from '@solidjs/start/config'

export default defineConfig({
  experimental: {
    islands: true,
  },
})
```

This is best when SolidStart is the application shell. For Jinja2 or JX-first
projects, the simpler pattern is usually plain Solid widgets mounted into
server-rendered templates.

## Boundaries and Caveats

- Do not assume router context or arbitrary shared context across island
  boundaries.
- Keep each island self-contained in props, state, and events.
- Pass serialized data through JSON script payloads or HTML attributes.
- Keep outer layout ownership on the server to avoid hydration mismatch.
- Use `<NoHydration>` only when a Solid-rendered region should stay static after
  server render; it is not a way to hydrate nested children later.

## Island Communication

If you have multiple independent islands on the same page that need to synchronize state (e.g., a "Shopping Cart" island and an "Add to Cart" island), avoid wrapping the entire page in a Solid context. Instead, use minimal decoupled state management:

1. **Custom Events:** Dispatch HTML `CustomEvent`s on `window` or `document` and listen for them inside your islands.
2. **Nanostores or atom-based stores:** Share a lightweight global state store (e.g., `@nanostores/solid`) across islands imported in the same bundle.

```tsx
// store.ts
import { atom } from 'nanostores'
export const cartCount = atom(0)

// Island A (updates state)
import { useStore } from '@nanostores/solid'
import { cartCount } from '../store'
cartCount.set(cartCount.get() + 1)

// Island B (reads state)
const count = useStore(cartCount)
```

## Tooling and Multiple Entry Points

When dealing with many islands in a Jinja/FastAPI setup, use Vite's multi-entry configuration so you only load the islands needed for a specific template.

```ts
// vite.config.ts
import { defineConfig } from 'vite'
import solidPlugin from 'vite-plugin-solid'
import { resolve } from 'path'

export default defineConfig({
  plugins: [solidPlugin()],
  build: {
    rollupOptions: {
      input: {
        'dashboard-islands': resolve(__dirname, 'src/entry-dashboard.tsx'),
        'catalog-islands': resolve(__dirname, 'src/entry-catalog.tsx')
      }
    }
  }
})
```

## Rules of Thumb

- Prefer multiple small islands over one page-sized island in Jinja or JX apps.
- Keep island props explicit and serializable.
- Put island bootstrapping in one client entry file per page or feature area.
- Let templates own markup semantics and let Solid own behavior.
